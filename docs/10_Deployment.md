# 10 — Deployment Architecture

## 10.1 Development Environment (Local)

**Prerequisites:** .NET 10 SDK, Node.js 18+, SQL Server LocalDB (or full SQL Server Developer Edition), Docker Desktop (optional, for Testcontainers-based integration tests).

**Setup steps:**
```bash
# Backend
cd backend
dotnet restore
dotnet ef database update --project src/LitXus.Infrastructure --startup-project src/LitXus.Api
dotnet run --project src/LitXus.Api          # https://localhost:5001, Swagger at /swagger

# Frontend
cd frontend
npm install
npm run dev                                   # http://localhost:5173, proxies /api to backend
```

`appsettings.Development.json`: `Seeding:Enabled=true`, connection string to LocalDB, Serilog console sink only, CORS open to `localhost:5173`.

## 10.2 Demo Environment (Cloud)

- Purpose: sales demos, UAT walkthroughs, prospective-customer trials.
- Hosted on Azure App Service (or AWS Elastic Beanstalk — either works, Azure preferred for tighter SQL Server integration) + Azure SQL Database (Basic/Standard tier — low traffic).
- Frontend built (`npm run build`) and served as static files from the same App Service as the API (`wwwroot`), avoiding a separate CDN/hosting bill for a low-traffic demo.
- `Seeding:Enabled=true` here too — demo environment should always show realistic sample data.
- Reset job (scheduled, e.g. nightly) re-seeds the demo database from scratch so prospect trials don't accumulate junk data indefinitely.

## 10.3 Production — Three Customer Options

**Option 1: Self-Hosted (customer's own server, IIS)**
- Customer runs Windows Server + IIS with the ASP.NET Core Hosting Bundle installed.
- Deployment: `dotnet publish -c Release`, output copied to IIS site folder, `web.config` generated by the SDK handles the reverse-proxy-to-Kestrel wiring.
- SQL Server: customer's own instance (on the same box or their existing SQL Server).
- Frontend: built static files served from `wwwroot` under the same IIS site (single site, single port, simplest for customer IT to firewall).

**Option 2: Cloud-Hosted (customer's own AWS/Azure account)**
- Same artifact (published backend + built frontend), deployed to the customer's cloud VM or App Service — LitXus Systems doesn't operate this infrastructure, just documents the deployment steps and provides a Docker image as the easiest path.

**Option 3: Managed Service (LitXus-operated, Malaysia-based cloud)**
- LitXus Systems hosts and operates the instance on the customer's behalf — same Docker image, deployed to a Malaysia-region cloud provider to satisfy data-residency preferences, with LitXus handling backups/updates/monitoring as a service.

All three options ship the **same build artifact** — the only difference is who operates the infrastructure.

## 10.3a Initial Setup (First Boot)

A fresh production install has **nothing seeded** — `Seeding:Enabled=false` means even the permission/role catalog would normally be skipped, except `RbacSeeder` runs in every environment regardless (it's reference data the app can't function without, not sample data — see `docs/08_Sample_Data.md` §8.5). Nothing else is seeded: no demo users, no demo company, no demo license, no demo accounting data.

To bring a fresh instance up:

1. **Keep the instance private** (firewalled / not yet publicly reachable) until step 2 is done — whoever completes step 2 first becomes the permanent install owner, and there's no UI/API path to demote or replace them afterward except direct database access.
2. Visit `/auth/register` and create the first account. It is automatically activated and granted the **Super Admin** role — no seeded account exists to log in with instead.
3. Log in, go to **Administration → License**, and apply the vendor-issued license key (generated offline via `backend/tools/LitXus.LicenseGenerator` — see `docs/17_License_Generator.md`). Nothing else in the app is usable until this step, since every module is gated behind an active license.
4. Go to **Administration → Company Profile** and fill in the real company's registration details — this feeds every financial report's letterhead.
5. Onboard additional Admin/User accounts directly — go to **Administration → Users → + New User**, enter their name/email/an initial password, and pick a role. The account is active immediately; there's no self-registration or separate activation step for anyone after the first bootstrap user (`/auth/register` rejects every call once any user exists). See `docs/phase-1-accounting/Admin_Setup_User_Guide.md` §3.

## 10.4 Docker Configuration

```dockerfile
# backend/Dockerfile (multi-stage)
FROM mcr.microsoft.com/dotnet/sdk:9.0 AS build
WORKDIR /src
COPY . .
RUN dotnet publish src/LitXus.Api -c Release -o /app

FROM mcr.microsoft.com/dotnet/aspnet:9.0 AS final
WORKDIR /app
COPY --from=build /app .
COPY --from=frontend-build /frontend/dist ./wwwroot
ENTRYPOINT ["dotnet", "LitXus.Api.dll"]
```

```yaml
# docker-compose.yml (self-hosted quick-start)
services:
  db:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      ACCEPT_EULA: "Y"
      MSSQL_SA_PASSWORD: "${DB_SA_PASSWORD}"
    volumes: [ "litxus-db-data:/var/opt/mssql" ]
  api:
    build: ./backend
    depends_on: [ db ]
    environment:
      ConnectionStrings__Default: "Server=db;Database=LitXus;User=sa;Password=${DB_SA_PASSWORD};TrustServerCertificate=true"
      Seeding__Enabled: "false"
    ports: [ "5000:8080" ]
volumes:
  litxus-db-data:
```

## 10.5 Database Backup / Recovery

- Self-hosted/managed: nightly `sqlpackage`/native SQL Server backup to a customer-controlled or LitXus-managed storage location, 30-day rolling retention minimum (separate from the 7-year AuditLogs retention, which lives inside the DB itself and is covered by the same backups).
- Point-in-time recovery enabled where the hosting tier supports it (Azure SQL, AWS RDS SQL Server).
- Restore procedure documented step-by-step in the deployment guide ([12_Documentation_Templates.md](12_Documentation_Templates.md)), tested at least once per phase-5-equivalent release cycle.

## 10.6 Configuration Management

- `appsettings.json` (defaults) → `appsettings.{Environment}.json` (overrides) → environment variables (secrets, connection strings — never committed) → this is standard ASP.NET Core config layering, no custom `.env` parser needed on the backend.
- Frontend: Vite `.env.production` / `.env.development` for `VITE_API_BASE_URL` only — no secrets on the frontend by design (JWT is the only credential, obtained via login, never baked into the build).
- Secrets in Production: environment variables injected by whichever hosting option is used (IIS app pool env vars, Docker secrets, or cloud provider's secret manager) — never in a committed file.

## 10.7 Monitoring & Logging

- Serilog: Console sink (structured JSON) always on; rolling file sink for self-hosted (customer's own server, for their IT to inspect); Application Insights or equivalent cloud sink for Demo/Managed-Service deployments.
- Correlation ID middleware tags every request/log line, so a support request ("invoice creation failed for user X at 3:15pm") can be traced through the full log chain.

## 10.8 Health Checks & Diagnostics

- `GET /health` (ASP.NET Core Health Checks middleware): DB connectivity, migration-up-to-date check.
- `GET /health/detailed` (admin-only): last successful seed run, current license status/expiry, enabled modules — useful first-line diagnostic for support calls.

## 10.9 Troubleshooting Guide (skeleton, filled in per phase)

Common issues to document as they're discovered during each phase's testing: migration failures on first run (usually SQL Server collation mismatch), JWT clock-skew errors (server time not synced), CORS errors in local dev (frontend/backend port mismatch), Docker Compose DB not ready before API starts (documented `depends_on` + healthcheck fix).
