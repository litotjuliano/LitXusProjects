## GL Entries (102 rows, built)

Generated by `AccountingDemoDataSeeder` across Jan–Jun {current year}, telling one coherent story: a Malaysia SME distributor with two ongoing customers (**Tropikal Hardware Sdn Bhd**, invoiced monthly and never paid down — an intentionally aging receivable — and **Selangor Pipe Supplies Sdn Bhd**, invoiced and paid within the same month), monthly rent/salaries/utilities/bank charges/office supplies, an equipment purchase, a partly loan-funded delivery van, insurance amortization, straight-line-ish depreciation, and periodic marketing/vehicle/interest costs.

| Status | Count |
|---|---|
| Posted | 93 |
| Draft | 4 |
| Voided | 5 |
| **Total** | **102** |

Specific edge-case rows required by [Test_Scenarios.md](Test_Scenarios.md), all present:
- 2 Draft entries that are internally balanced ("pending approval" style — never posted by the seeder itself)
- 2 Draft entries that are **intentionally unbalanced**, so attempting to post one through the UI has something real to reject (verified: returns `422 ENTRY_UNBALANCED` with the correct delta)
- 5 Voided entries, each posted then voided with a real reason (e.g. "Duplicate entry — see the actual January rent payment entry"), entry numbers retained and never reused
- 1 zero-value entry (both lines RM 0.00) — posts successfully, confirming the balance check doesn't false-negative on zero
- 1 future-dated entry (September, beyond the "current" June data) — posts successfully, since Phase 1 has no period-close to reject it

**Verified balanced**: Trial Balance and Balance Sheet both sum to RM 194,114 / RM 103,846 respectively (both sides equal) as of a date capturing every entry including the future-dated one.

## Bank Accounts / Bank Statement Lines — not yet seeded

The original plan called for 2 bank accounts (Maybank, CIMB) and ~15 statement lines each with ~80% pre-matched. The `BankAccount`/`BankStatementLine` domain entities already exist (from the initial Phase 1 scaffold), but there's no API or UI for Bank Reconciliation yet (Phase 1 backlog item #4) — seeding this data now would just be invisible rows nobody can interact with. This will be seeded alongside that feature, not before.

## Seeder Implementation (as actually built)

`AccountingDemoDataSeeder : ISeeder`, `Order = 4`, runs after `RbacSeeder` (1) → `LicenseSeeder` (2) → `UserSeeder` (3). Two-stage idempotency: `SeedAccountsAsync` checks each `Account.Code` individually (so it layers safely on top of accounts a user already created by hand, rather than bailing out if *any* account exists), while `SeedGLEntriesAsync` checks `if (await db.GLEntries.AnyAsync()) return;` once, since the entry set is generated as one coherent whole and partial re-seeding wouldn't make sense. See [08_Sample_Data.md](../08_Sample_Data.md) §8.5 for how this fits into the overall seeding pipeline.
